# Sliding Window OR

## Problem

- **Source:** [CSES 3405: Sliding Window OR](https://cses.fi/problemset/task/3405/)
- **Code:** [`View accepted C++ solution (or.cpp)`](./or.cpp)
- **Limits:** $1\le k\le n\le10^7$, $0\le x,a,b\le10^9$, and $1\le c\le10^9$.

The sequence is generated by

$$
x_1=x,
\qquad
x_i=(ax_{i-1}+b)\bmod c
$$

for $i=2,3,\ldots,n$.

For every length-$k$ window, compute the bitwise OR of its elements. Output the bitwise XOR of all $n-k+1$ window ORs.

## Idea

Computing the OR of every window independently takes $O(nk)$ time. Since both $n$ and $k$ may be large, each generated value must be processed only a constant number of times.

For sums and XORs, an outgoing value can be removed using an inverse operation. Bitwise OR has no inverse. Knowing $u\mathbin{|}v$ does not determine the result after removing $u$.

Frequency counters for the individual bits would support each update in $O(1)$ time because there are about $30$ relevant bits, but they would still perform roughly $30n$ bit operations. Instead, represent the window as a queue consisting of two aggregate stacks.

Each stack stores pairs

$$
(\text{value},\text{aggregate}),
$$

where `aggregate` is the OR of that entry and every entry below it. For a stack containing $y_1,y_2,\ldots,y_t$ from bottom to top, its top aggregate is

$$
y_1\mathbin{|}y_2\mathbin{|}\cdots\mathbin{|}y_t.
$$

The queue consists of:

- `s2`, which contains the oldest values with the queue front at its top;
- `s1`, which contains the newest values with the newest value at its top.

Therefore, the OR of the entire queue is

$$
\operatorname{OR}(\texttt{s2})\mathbin{|}\operatorname{OR}(\texttt{s1}),
$$

ignoring an empty stack.

New values are added to `s1`. To delete the oldest value, pop from `s2`. If `s2` is empty, move every value from `s1` to `s2`. This reverses their order and places the oldest value on top of `s2`.

One transfer may move many entries, but every entry is moved at most once. The total work therefore remains linear.

The generator arithmetic uses `long long` because $ax+b$ may approach $10^{18}$.

## Algorithm

Maintain two aggregate stacks, `s1` and `s2`, and the current queue size `sz`.

For every zero-based index $i$ from $0$ to $n-1$:

1. Add the current generated value to `s1`. Its aggregate is the OR of its value and the previous aggregate of `s1`.
2. If the queue now contains more than $k$ values:
   - If `s2` is empty, move every value from `s1` to `s2` and recompute the aggregates.
   - Pop from `s2`, removing the oldest value.
3. If $i\ge k-1$, OR the top aggregates of `s1` and `s2`, then XOR this window OR into `ans`.
4. Advance the generator.

Output `ans`.

## Correctness

We prove that the algorithm outputs the XOR of all length-$k$ window ORs.

**Lemma 1.** For either stack, the aggregate stored at its top equals the OR of all values in that stack.

**Proof.** When a value $v$ is inserted into an empty stack, its aggregate is $v$. Otherwise, let $A$ be the previous top aggregate. By induction, $A$ is the OR of all previous stack values. The new aggregate is $A\mathbin{|}v$, which is the OR of the entire stack after insertion. $\square$

**Lemma 2.** The two stacks represent the queue in the correct insertion order.

**Proof.** New values are pushed onto `s1`. When `s2` is empty, repeatedly popping from `s1` and pushing onto `s2` reverses their order. The oldest value, previously at the bottom of `s1`, becomes the top of `s2`. Therefore, popping from `s2` removes the oldest queue value. $\square$

**Lemma 3.** Whenever the queue contains the current window, the OR computed from the two top aggregates equals the OR of that window.

**Proof.** By Lemma 1, each top aggregate is the OR of its stack. By Lemma 2, `s1` and `s2` partition the values of the queue. Therefore,

$$
\operatorname{OR}(\text{window})
=
\operatorname{OR}(\texttt{s1})\mathbin{|}\operatorname{OR}(\texttt{s2}),
$$

which is exactly the value computed by the algorithm. $\square$

**Theorem.** The final value of `ans` is the XOR of all length-$k$ window ORs.

**Proof.** After processing index $i$, the algorithm removes the oldest value whenever the queue size exceeds $k$. Thus, for every $i\ge k-1$, the queue contains exactly the length-$k$ window ending at $i$. By Lemma 3, the computed aggregate is its exact OR. The endpoints $k-1,k,\ldots,n-1$ correspond to all $n-k+1$ windows, and each result is XORed into `ans` once. $\square$

## Implementation

Each stack entry is a `pair<int, int>`:

```cpp
vector<pair<int, int>> s1, s2;
```

The first component stores the value, which is needed when transferring an entry from `s1` to `s2`. The second component stores the aggregate, allowing the OR of an entire stack to be retrieved in constant time.

During a transfer, aggregates must be recomputed in the new stack order:

```cpp
z = y;
if (!s2.empty()) {
    z |= s2.back().second;
}
```

The current value is inserted before the queue is reduced to size $k$. If its size becomes $k+1$, the oldest value is removed, leaving exactly the current window.

The window OR is obtained from the two top aggregates:

```cpp
int cur = 0;
if (!s1.empty()) {
    cur |= s1.back().second;
}
if (!s2.empty()) {
    cur |= s2.back().second;
}
```

Individual generated values and window ORs are below $2^{30}$, so `int` is sufficient for stack values and `ans`. The generator parameters use `long long` to prevent overflow during multiplication.

### Accepted C++17 implementation

```cpp
#include <bits/stdc++.h>
using namespace std;

using ll = long long;

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n, k;
    ll x, a, b, c;
    cin >> n >> k >> x >> a >> b >> c;

    vector<pair<int, int>> s1, s2;
    int sz = 0, ans = 0;

    for (int i = 0; i < n; i++) {
        int y = x;
        int z = y;
        if (!s1.empty()) z |= s1.back().second;
        s1.push_back({y, z});
        sz++;

        if (sz > k) {
            if (s2.empty()) {
                while (!s1.empty()) {
                    y = s1.back().first;
                    s1.pop_back();

                    z = y;
                    if (!s2.empty()) z |= s2.back().second;
                    s2.push_back({y, z});
                }
            }

            s2.pop_back();
            sz--;
        }

        if (i >= k - 1) {
            int cur = 0;
            if (!s1.empty()) cur |= s1.back().second;
            if (!s2.empty()) cur |= s2.back().second;
            ans ^= cur;
        }

        x = (a * x + b) % c;
    }

    cout << ans << '\n';
}
```

## Complexity

Each generated value is pushed onto `s1` once, transferred from `s1` to `s2` at most once, and popped from `s2` at most once. The total time complexity is $O(n)$.

The two stacks together contain at most $k$ current-window values after each update, so the auxiliary space complexity is $O(k)$.

## Worked Example

For $n=8$, $k=5$, $x=3$, $a=7$, $b=1$, and $c=11$, the generated sequence is

$$
[3,0,1,8,2,4,7,6].
$$

| Window | Values | Window OR | XOR answer |
|---:|---|---:|---:|
| $1$ | $[3,0,1,8,2]$ | $11$ | $11$ |
| $2$ | $[0,1,8,2,4]$ | $15$ | $4$ |
| $3$ | $[1,8,2,4,7]$ | $15$ | $11$ |
| $4$ | $[8,2,4,7,6]$ | $15$ | $4$ |

The final answer is $4$.
